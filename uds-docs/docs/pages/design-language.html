Design Language

How UDS states actually look, beyond the token vocabulary. Every visual treatment on this page is derived from the existing component CSS — when components disagree, both patterns are documented and the disagreement is surfaced at the bottom for designer ratification.

How to read this page. Each section starts with a short description, then a table mapping every component to how it implements that pattern. Disagreements between components are surfaced inline AND in the Known inconsistencies section at the bottom. The source of truth for each row is the component's CSS file under uds-docs/uds/components/<id>/<id>.css.

1.4 Focus (keyboard-only)

The most important state to get right because UDS uses three distinct constructions, and each has theme-safety implications. See section 6 for the full deep dive.

Rule: use :focus-visible, never :focus. The latter triggers on every click and produces visual noise. :focus-visible only fires on keyboard focus, which is when users actually need the ring.

3. Availability states

3.1 Disabled

Non-interactive. Pointer events blocked, cursor not-allowed. UDS has three distinct disabled patterns depending on the element family.

PatternWhen usedTreatment
Filled-disabledPrimary / secondary buttons, form fields with a backgroundbg → surface-interactive-disabled, border → transparent OR border-disabled inconsistent, text → text-disabled-bold
Outlined-disabledForm controls (checkbox, radio)border → border-disabled, no bg change, label → text-disabled
Text-only-disabledGhost button, link, tab, toggle labeltext → text-disabled, pointer-events: none, cursor: not-allowed
Reduced-opacity-disabledToggle (only)border → border-disabled, opacity: 0.7 outlier
Disabled text token nuance: use text-disabled for general copy and text-disabled-bold for the value a user typed into a form field. The "-bold" variant has slightly higher contrast so a user's entered text stays legible even when greyed.

3.2 Read-only

Looks like a field but accepts no edits. Implemented inconsistently today — only partially supported in code. The Text Input draft in the Figma factory is the first explicit treatment.

3.3 Loading

Not specified in UDS today. Components that need it currently fall back to Disabled as the closest approximation (see text-input.css known issues). Deferred for designer ratification.

5. Disabled vs. Read-only — quick reference

These two are regularly confused. Use this table to disambiguate.

QuestionDisabledRead-only
User can read the value?technically yes, but greyedyes, full primary text
User can copy the value?usually not (cursor not-allowed)yes
User can focus the element?no (skipped by Tab)yes
User can edit?nono
Communicates "temporarily unavailable"?yesno — communicates "this is fixed information"
Communicates "you can't change this"?yes (because it's broken / not yet allowed)yes (because the system owns it)
Common confusion. Applying Disabled to "this field is auto-filled and shouldn't be edited" — that's Read-only. Disabled is for the action isn't currently available (e.g. a "Save" button while the form is invalid).

6. Focus ring construction (deep dive)

UDS has three distinct focus-ring constructions. They are not interchangeable. Pick by the rules below, not by aesthetic preference.

6.1 Construction A — outline + outline-offset recommended default

outline: 2px solid var(--uds-color-border-outline-focus-visible); outline-offset: 1px; /* or 2px, or var(--uds-space-050) for chunkier */

Used by: checkbox, radio, link, toggle, tabs, text-area, dropdown items, dialog close button, notification close button, every trailing icon button inside a field.

Why it works: outline-offset creates a transparent gap. It matches whatever's behind the element in any theme automatically. The ring color is the only colored thing.

When to use: every new focus ring should use this construction unless there's a specific reason not to.

6.2 Construction B — box-shadow stacked with surface fill

box-shadow: 0 0 0 2px var(--uds-color-surface-main), 0 0 0 4px var(--uds-color-border-outline-focus-visible);

Used by: chip, search.

When the inner shadow is bound to surface-main: theme-safe — the inner ring matches whatever surface the element sits on. In any theme it visually blends with the surrounding surface and reads as a gap.

When the inner shadow is bound to surface-white: theme-fragile — surface-white is white in every theme. In dark mode you get a visible white ring instead of a gap. This is the bug in button.css, text-input.css, and dropdown.css today.

When to use B instead of A: when the element sits inside another bordered container that would clip an outline (the field's overflow: hidden breaks outline rendering). The stacked box-shadow doesn't get clipped.

Hard rule: if you use construction B, the gap layer MUST be bound to surface-main (or whatever surface the element actually sits on), NEVER surface-white. The current surface-white usage in button, text-input, and dropdown is a documented bug — see section 12.1.

6.3 Construction C — box-shadow single ring, no gap

box-shadow: 0 0 0 3px var(--uds-color-border-outline-focus-visible);

Used by: tile.

Why it works: when the element is a large card or tile with its own visual weight, a direct ring on the element reads cleanly without a gap to disambiguate from neighbors.

When to use: rarely. Only when the element is a large container whose border or content would compete visually with a gapped ring.

6.4 Decision tree

Is the element inside a clipped container (overflow:hidden)? ├── Yes → Use construction B (box-shadow stacked) bound to surface-main └── No │ Is the element a large card / tile? ├── Yes → Use construction C (single ring, no gap) └── No → Use construction A (outline + offset) ← default

6.5 Universal rules

  • The ring color is always border-outline-focus-visible (blue).
  • Ring is 2 px thick for inline controls, form fields, and form controls.
  • Ring is 3 px thick for tiles (single-ring construction).
  • Gap (in A or B) is 1–2 px.
  • Uses :focus-visible, never :focus.
  • Error doesn't recolor the focus ring — it stays blue (search.css is the only exception today; see section 12).

8. Elevation / shadow scale

Three depth levels, plus a special bento variant for dashboards. Pick by the element's purpose, not its size.

TokenResolved value (light theme)Use when…Live examples
shadow-depth-1000 2px 8px rgba(0, 0, 0, 0.1)A small element is lifted off its surfaceToggle thumb
shadow-depth-3000 4px 12px rgba(0, 0, 0, 0.08)A transient overlay near its triggerTooltips, dropdown menus, popovers
shadow-depth-5000 10px 20px 2px rgba(0, 0, 0, 0.2)A modal that floats above the entire pageDialogs, sheets
shadow-bento0 6px 16px 1px rgba(0, 0, 0, 0.16)Bento-style card grouping (dashboard layouts)Dashboard cards

Backdrop overlay (under modals): --uds-overlay-backdrop resolves to rgba(0, 0, 0, 0.45).

Don't bind elevation tokens to non-elevation effects. The shadow scale exists to communicate depth, not decoration. A "drop shadow on a button to make it look fancy" is design noise — the scale ends at three deliberate levels.

11. Selection vs. Checked vs. Active — naming canon

Three words that are often confused. Use the right one for the right element.

TermUsed byWhat it means
Checkedcheckbox, radio, toggleNative HTML form-control selection state. Pair with <input checked> / aria-checked="true".
Selectedtabs, dropdown items, chip, tileNon-form-control selection state. Pair with aria-selected="true" or data-selected="true".
Active (as in current)link, button ([data-selected])"This is the current page / navigation target." Pair with aria-current="page".
Active (as in pressed)every interactive control during :activeMouse-down state. Brief, ends on mouse-up.
The two meanings of "Active" collide. When writing a spec or rule, prefer "Pressed" for the mouse-down meaning and "Current" for the selected-as-nav meaning. The CSS uses :active for pressed and aria-current for current.

Cross-cutting decisions

  • Stroke alignment is INSIDE for form fields so border thickness doesn't affect outer dimensions. Form controls (checkbox, radio) use default stroke alignment.
  • Border weight stays at 1 px through state transitions for most components. Exceptions: tile thickens to 2 px when selected; tabs use a 2 px bottom border for selection.
  • The :focus-visible pseudo-class is the focus selector, never :focus (which fires on every click). All UDS components use :focus-visible.
  • Transitions are 120ms ease for color and border changes, 200ms ease for transforms (the dropdown chevron rotation, the toggle thumb slide).
  • Disabled elements have pointer-events: none AND cursor: not-allowed. Both are required: pointer-events blocks click, cursor communicates the state to the user before they try.
