Cursor Workflows

UDS uses three layers of Cursor agent guidance to keep designer-led changes safe, repeatable, and shareable. All three are version-controlled in this repo, so anyone cloning gets them automatically.

Overview

Three concepts, ordered from "always on" to "called only when needed":

TypeWhat it isLives in
Rule Standing instruction. Always on (or scoped by file globs). The agent reads it on every task.
Analogy: "Always wear a seatbelt."		 .cursor/rules/*.mdc
Skill How-to procedure. Loaded on demand when the task description matches. The agent itself follows the steps.
Analogy: "How to change a tire — read the manual when you need it."		 .cursor/skills/<name>/SKILL.md
Subagent Specialist contractor. A new agent spawned in its own context, does its work, returns a result, then disappears.
Analogy: "Call the mechanic."		 .cursor/agents/<name>.md

Figma-to-docs workflow

The prompt UDS updated is the top-level Figma sync workflow. It reads UDS Tokens and UDS Components, compares against tracked sync snapshots and the doc site, classifies changes by confidence and breaking risk, then applies only safe high-confidence updates. Token ZIP exports are fallback only — direct Figma Variables reads are the primary source. Figma pages with {Ignore} anywhere in the page name are skipped for UDS sync, inventory, status, spec, and release workflows. Figma checks are read-only in behavior, but must run with MCP-enabled tool access — do not run them in an execution mode that blocks MCP reads. Release changelogs must include a component-set variant coverage pass for every non-ignored status-prefixed Figma component page; page names alone are not enough.

StepWhat happensSafety rule
Capability checkConfirms the Figma tool can read pages, Variables collections, modes, node trees, bound variables, and release-note write targets.Fail closed if required Figma data is unavailable.
Token auditReads the UDS Tokens Variables tab directly, validates the token architecture, normalizes tokens, and diffs against CSS + the last sync snapshot.ZIPs are requested only when direct variable reads fail or are incomplete.
Component inventoryReads UDS Components pages, stoplight prefixes, component sets, variant-property matrices, and page fingerprints to find new, moved, deleted, status-changed, or variant-changed components.Release notes cannot be written until every non-ignored component set has variant coverage captured.
Component inspectionFor each changed component, reads node trees, layer names, variants, nested instances, token bindings, auto-layout, anatomy, and doc-site mismatches.Screenshots are supporting evidence only; never enough to update specs.
Safe applyApplies only high-confidence, non-breaking changes; reports medium/low confidence and breaking changes for review.Never auto-delete tokens or components. Preserve/deprecate/report first.

Twelve active rules. The first six protect the doc site; the Figma rules make token/component sync direct-read, deep-inspection, confidence-scored, and deletion-safe.

Active Skills

Skills load when the agent sees a task description that matches.

new-component

Scaffold a new UDS component end to end. Creates content/<id>.json with required-minimum fields, an empty CSS file, sidebar entry, page section, COMPONENT_STATUS entry, and bumps the SITE version. Component lands at "placeholder" status with ~6/22 spec completeness — designer fills in the rest over time.

Triggers: "add a new component", "scaffold a component called X", "set up a new component for Y"

File: .cursor/skills/new-component/SKILL.md

Active Subagents

Subagents spawn in an isolated context window and return a single result. They keep noisy intermediate work (file reads, parallel scans) out of the main conversation.

spec-audit read-only

Audits content/*.json spec completeness across one or all UDS components. Reports gaps and recommends the highest-impact fields to fill in next. The same scoring logic the "Spec X/22" pill uses on every component page.

Triggers: "audit our specs", "which components are incomplete?", "audit the X spec", "spec audit before release"

Returns: Markdown report with aggregate stats (average completeness, list below 50%, list missing required fields) and per-component findings (score, missing required, top 3 recommended next fields).

File: .cursor/agents/spec-audit.md

How to add new ones

Add a Rule — drop a .mdc file in .cursor/rules/ with frontmatter (description, alwaysApply or globs) and the rule body in markdown. Use rules for standing instructions and project conventions ("always do X", "never use Y").

Add a Skill — create .cursor/skills/<name>/SKILL.md with frontmatter (name, description) and a step-by-step procedure. Optionally add scripts/, references/, or assets/ subfolders. Use skills for repeatable, well-defined procedures the agent can carry out itself ("scaffold X", "release Y").

Add a Subagent — create .cursor/agents/<name>.md with frontmatter (name, description, optionally readonly: true, model, is_background) and the subagent's instructions. Use subagents for tasks that benefit from context isolation: long research, parallel work, or read-only audits.

All three live in the repo and ship with anyone who clones it. The How to Contribute page covers when to reach for which.

Why this matters for designers

A designer doesn't need to memorize how to scaffold a component, audit specs, sync changelogs, or run a release. Saying "scaffold a new tooltip component" or "audit our specs" is enough — Cursor picks the right skill or subagent based on the description, and the rules guarantee the version bump and changelog land too. Less ceremony, fewer mistakes, more time on actual design.
