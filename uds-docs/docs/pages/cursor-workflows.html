Overview

Four kinds of guidance, ordered from "always on" to "called only when needed" to "enforced by CI":

TypeWhat it isLives in
Rule Standing instruction. Always on (or scoped by file globs). The agent reads it on every relevant task.
Analogy: "Always wear a seatbelt."		 .cursor/rules/*.mdc
Skill How-to procedure. Loaded on demand when the task description matches. The agent itself follows the steps.
Analogy: "How to change a tire — read the manual when you need it."		 .cursor/skills/<name>/SKILL.md
Subagent Specialist contractor. A new agent spawned in its own context, does its work, returns a result, then disappears.
Analogy: "Call the mechanic."		 .cursor/agents/<name>.md
Audit Bash script invoked by CI on every PR + push to main. Failing audits block merge. Catches the failure modes the rules describe but humans / agents skip.
Analogy: "The MOT inspection — if the seatbelt's missing, the car doesn't pass."		 scripts/audit-*.sh

Workflow diagrams

Four end-to-end workflows the agent toolchain supports. Each shows the order of steps, what each step touches, and which skills / subagents / rules / audits are involved. The orchestrator skills (uds-updated, figma-component-card) tie multiple steps together; the specialist sync skills can also be invoked individually.

1. UDS version release (Figma X.Y → docs site)

Triggered when Figma's UDS Version: X.Y page bumps and the user confirms a release. Orchestrated by uds-updated or the release questionnaire in uds-release-workflow.mdc.

1
Preflight — figma-capability-check + figma-inventory
Confirms Figma MCP can read pages / Variables / node trees / bound variables, then walks both UDS files page-by-page to find the version bump, status changes, new components, and renamed/moved components.
Reads: both Figma files, uds/version.json, uds/components.json, every status.json. Writes: nothing.
↓
2
Token audit — figma-token-audit
Direct Variables read of the UDS Tokens file. Reports added / removed / renamed tokens, value changes per mode, alias-graph validity, and whether ZIP fallback is needed. Architecture validated against uds-token-architecture.mdc.
Reads: UDS Tokens Figma, uds/tokens/*.css, tokens.snapshot.json. Writes: nothing.
↓
3
Component inspections — figma-component-inspector (one per changed component)
For each component the inventory flagged as changed, performs a deep node-tree + variant-matrix + token-binding + nested-instance + accessibility read. Compares against the per-component spec.json / <id>.css / examples/*.html / impl.json / playground.js.
Reads: UDS Components Figma, every uds/components/<id>/ file. Writes: nothing.
↓
4
Snapshot release — release.sh <new>
Archives the current uds/ into versions/<old>/uds/; bumps uds/version.json; updates versions.json; runs aggregate-changelog.sh and aggregate-components.sh.
Writes: versions/<old>/uds/, uds/version.json, versions.json, uds/CHANGELOG.json, uds/components.json.
↓
5
Apply changes — sync-figma-component-status, sync-figma-component-spec, link-figma-nodes, import-figma-tokens, new-component
High-confidence changes apply per uds-figma-change-classification.mdc. Each skill writes per-component changelog entries + updates .cursor/figma/state/components.snapshot.json. Token import regenerates token CSS from direct Variables reads (ZIPs are fallback only).
Writes: uds/components/<id>/{spec,status,changelog,impl}.json, <id>.css, <id>.js, examples/*, playground.js; uds/tokens/*.css; .cursor/figma/state/*.json.
↓
6
Aggregate + re-audit — aggregate-changelog.sh, aggregate-components.sh, audit-theme-contrast.sh
Rebuilds uds/CHANGELOG.json and uds/components.json from per-component files; runs the 6-theme contrast audit to catch any regression in the token-import.
Writes: uds/CHANGELOG.json, uds/components.json. Reads: all per-component files + token CSS.
↓
7
Figma Release Notes sync — sync-figma-release-notes
Removes the existing Release Notes frame on the UDS Version: X.Y page in both UDS Tokens and UDS Components Figma files and rebuilds from the aggregated uds/CHANGELOG.json. Both files must end up with identical text.
Writes: Release Notes frame in BOTH Figma files. Allowed write scope #1 per uds-figma-write-safety.mdc.
↓
8
SITE bump + commit + push — bump-site.sh
Bumps version.txt + inline SITE_VERSION + display string. Appends a SITE_CHANGELOG entry. Cache-busts uds.css?v=, uds.js?v=, app.js?v= for the files that changed. Pushes to main; GitHub Actions deploys to GH Pages.
Writes: uds-docs/version.txt, uds-docs/index.html, uds-docs/docs/data/site-changelog.js.
End-to-end this workflow touches: every file under uds/components/<id>/ for changed components, all uds/tokens/*.css, uds/CHANGELOG.json, uds/components.json, uds/version.json, versions.json + a new versions/<old>/uds/ snapshot, .cursor/figma/state/{components,tokens}.snapshot.json, .cursor/figma/state/last-sync.json, the Release Notes frames in BOTH Figma files, uds-docs/index.html, uds-docs/version.txt, uds-docs/docs/data/site-changelog.js.

Audits that gate the merge: audit-component-completeness, audit-placeholders, audit-token-usage, audit-demo-coverage, audit-changelog-currency, audit-aggregate-currency, audit-figma-sync-state-currency, audit-css-api-table, audit-agent-docs-currency, audit-theme-contrast (after token regen).

2. Mid-release component sync (e.g. "sync nav-header from Figma")

No version bump — one or a few components get a fresh Figma read and the doc-site catches up. Triggered by a prompt like "sync nav-header from Figma" or "the title-area component renamed in Figma." The failure modes addressed by PR2's new audits all live in this loop.

1
Preflight + deep inspect — figma-component-inspector
Per uds-figma-component-inspection.mdc: locate canonical component set, walk node tree, extract variant matrix + token bindings + nested instances, classify findings per uds-figma-change-classification.mdc. Screenshots are supporting evidence only.
Reads: UDS Components Figma; uds/components/<id>/*; .cursor/figma/state/components.snapshot.json. Writes: nothing.
↓
2
Apply spec sync — sync-figma-component-spec (apply mode)
For each high-confidence non-breaking finding: update spec.json (props, events, slots, states, accessibility, knownIssues, figmaNodeId), <id>.css (token bindings, drift fixes), examples/*.html (markup that matches the canonical anatomy), impl.json (Code-tab reference data), playground.js (controls + render). Potentially-breaking changes stop for explicit user direction.
Writes: every uds/components/<id>/ file that the spec touches.
↓
3
Per-component changelog + aggregate
Add an entries[] item to uds/components/<id>/changelog.json for the current UDS version — one entry per logical change (figmaNodeId fix, spec sync, rename, drift fix, etc.). Run aggregate-changelog.sh. audit-changelog-currency will fail CI if this step is skipped; audit-aggregate-currency will fail if you skip the aggregate.
Writes: uds/components/<id>/changelog.json, uds/CHANGELOG.json.
↓
4
Update Figma sync-state snapshot
Record the captured Figma node IDs, variant matrix, nested instances, and fingerprint into .cursor/figma/state/components.snapshot.json. Bump last-sync.json lastSuccessfulSync and recompute components.snapshotChecksum. audit-figma-sync-state-currency blocks merge if spec.json's figmaNodeId doesn't match the snapshot's componentSetNodeId.
Writes: .cursor/figma/state/components.snapshot.json, last-sync.json.
↓
5
Figma Release Notes sync — sync-figma-release-notes
uds-release-workflow.mdc "Changelog Sync Rule" mandates this whenever uds/CHANGELOG.json changes, even mid-release. Rebuilds the Release Notes frame on the current UDS Version: X.Y page in both Figma files.
Writes: Release Notes frame in BOTH Figma files.
↓
6
SITE bump + commit + push
bump-site.sh, append SITE_CHANGELOG entry describing the sync, bump cache-busters (uds.css?v= if component CSS changed; uds.js?v= if component JS changed; app.js?v= if the imported site-changelog.js changed). Push to main.
Writes: version.txt, index.html, site-changelog.js.
End-to-end this workflow touches: one or a few uds/components/<id>/ folders (the components being synced), uds/CHANGELOG.json, .cursor/figma/state/components.snapshot.json, .cursor/figma/state/last-sync.json, the Release Notes frames in BOTH Figma files, uds-docs/index.html, uds-docs/version.txt, uds-docs/docs/data/site-changelog.js.

The four PR2 audits (audit-changelog-currency, audit-aggregate-currency, audit-figma-sync-state-currency, audit-css-api-table) all hard-fail if any of these steps is skipped or partially completed.

3. Adding a new component

Triggered by prompts like "add a tooltip component," "scaffold a new pagination component." The new-component skill handles the scaffold end to end.

1
Scaffold — new-component skill
Creates uds/components/<id>/ with the required minimum files: <id>.css (empty token-first stub), spec.json (~6/22 completeness fields filled), status.json (initial status, usually placeholder or blocked), changelog.json (with a { version: "<current UDS>", type: "added" } entry), impl.json, playground.js default-export stub, examples/manifest.json + one example HTML file.
Writes: the entire new uds/components/<id>/ folder.
↓
2
Wire into the docs site
Adds the sidebar link in index.html under the right group; adds a placeholder <div data-page="<id>" data-page-fragment></div> OR an inline component page block with empty data-bind="title" / data-bind="description" placeholders (per the Phase 19 page-chrome refactor). Includes the component's CSS import in uds/uds.css; if interactive, registers the JS in uds/uds.js COMPONENT_SCRIPTS and the init selector.
Writes: uds-docs/index.html, uds/uds.css, uds/uds.js (if interactive).
↓
3
Refresh manifest + aggregated changelog
Runs aggregate-components.sh to add the new component to uds/components.json. Runs aggregate-changelog.sh to fold the component's initial "added" entry into uds/CHANGELOG.json.
Writes: uds/components.json, uds/CHANGELOG.json.
↓
4
SITE bump + commit + push
bump-site.sh + SITE_CHANGELOG entry announcing the new component + cache-bust uds.css?v= (because the new component's CSS is imported) and app.js?v= (because site-changelog.js changed). Push to main.
Writes: version.txt, index.html, site-changelog.js.
End-to-end this workflow touches: a brand-new uds/components/<id>/ folder + uds/uds.css + (optionally) uds/uds.js + uds/components.json + uds/CHANGELOG.json + uds-docs/index.html + uds-docs/version.txt + uds-docs/docs/data/site-changelog.js.

Status starts at placeholder or blocked; spec completeness rises over time as the designer fills in fields. No Figma Release Notes sync is required unless this component's addition lands as part of a UDS version bump.

4. Single component edit (e.g. "fix the bento chevron color")

Smallest scope — one component's CSS, examples, or playground gets adjusted (often as a follow-up to a Figma comment or an a11y review). The round-trip checklist still applies because per-component changelog.json is mandatory.

1
Source-of-truth check
uds-source-of-truth.mdc: any edit under uds-docs/uds/ must be sourced from Figma (or be a revert toward Figma parity). Cite the Figma read the edit is based on; never apply an audit-detected "fix" autonomously.
Reads: the Figma source (UDS Components or UDS Tokens). Writes: nothing yet.
↓
2
Edit the source file
Change exactly the targeted file in uds/components/<id>/. Token-first CSS only — no hardcoded colors / fonts / spacing (audit-token-usage enforces). If the edit changes CSS class names, the <table class="sg-api-table"> in index.html for that component must also update (audit-css-api-table enforces).
Writes: one file under uds/components/<id>/; potentially index.html.
↓
3
Per-component changelog + aggregate
Same as workflow 2 step 3 — add an entries[] item describing the fix, run aggregate-changelog.sh. The only exemption is a pure revert toward Figma parity — include [no-changelog: revert toward Figma parity] in the commit message and the audit will skip.
Writes: uds/components/<id>/changelog.json, uds/CHANGELOG.json.
↓
4
SITE bump + commit + push
Same SITE flow as the larger workflows. The Figma Release Notes sync (workflow 2 step 5) only runs if uds/CHANGELOG.json actually changed — for a small fix that's still yes.
Writes: version.txt, index.html, site-changelog.js.
End-to-end this workflow touches: one source file under uds/components/<id>/ + that component's changelog.json + uds/CHANGELOG.json + the SITE trio (version.txt, index.html, site-changelog.js). If you skip the per-component changelog entry, audit-changelog-currency blocks the merge.

Active Skills

All sync skills end their "Apply mode" by pointing at uds-master-preflight.mdc Phase 3 — the round-trip checklist lives in exactly one place.

Orchestrators (2)

uds-updated

End-to-end Figma → docs sync. Reads both Figma files, classifies findings, applies high-confidence non-breaking changes, falls back to dry-run / reporting for everything else.

figma-component-card

Builds the canonical seven-card layout on a UDS Components Figma page from uds/components/<id>/spec.json data. Direction of data flow is code → Figma (opposite of every other Figma-sync skill).

Calls: figma-capability-check, figma-component-inspector (for verification), figma-component-card-audit (subagent that verifies card structure post-write).

Baselines and exemptions

scripts/audit-baseline.json is the single config file for any audit that needs to grandfather pre-existing drift. It holds:

  • audit-changelog-currency — a baseline SHA. Commits before this SHA are not checked. Update only on a deliberate sweep that backfills changelogs.
  • audit-css-api-table — a toleratedComponents array. Components in this list are exempted from the Code-tab table check. Add/remove only with explicit user direction.
  • audit-agent-docs-currency — a baseline SHA and an optional toleratedFiles array for internal-only rules / skills / agents that don't need to appear on this docs page.

How to add new ones

Add a Rule — drop a .mdc file in .cursor/rules/ with YAML frontmatter (description, alwaysApply: true OR globs: [...]) and the rule body in markdown. Use rules for standing instructions and project conventions ("always do X," "never use Y"). Add a row to this page's Rules tables. audit-agent-docs-currency.sh blocks the merge otherwise.

Add a Skill — create .cursor/skills/<name>/SKILL.md with frontmatter (name, description) and a step-by-step procedure. Optionally add scripts/, references/, or assets/ subfolders. Use skills for repeatable, well-defined procedures the agent can carry out itself ("scaffold X," "sync Y"). Every sync skill should end with a one-line pointer to uds-master-preflight.mdc Phase 3 (the canonical round-trip checklist). Add a card or row to this page's Skills section.

Add a Subagent — create .cursor/agents/<name>.md with frontmatter (name, description, optionally readonly: true, model). Use subagents for tasks that benefit from context isolation: long research, parallel work, read-only audits. Add a row to this page's Subagents table.

Add an Audit — create scripts/audit-<name>.sh following the patterns of the existing audits (exit non-zero with a clear "FAIL — <reason>" line on the failure path; emit "OK — <summary>" on success). Wire it into the audits job in .github/workflows/audits.yml. If the audit needs grandfathering, extend scripts/audit-baseline.json. Add a row to this page's Audits table.

All four kinds live in the repo and ship with anyone who clones it. The How to Contribute page covers when to reach for which.

Why this matters for designers

A designer doesn't need to memorize how to scaffold a component, audit specs, sync changelogs, run a release, or update the doc-site chrome. Saying "sync the bento dropdown from Figma," "audit our specs," or "scaffold a new tooltip component" is enough — Cursor picks the right skill or subagent based on the description, the rules guarantee the version bump / per-component changelog / aggregate / snapshot / Figma Release Notes / cache-bust all land, and the CI audits hard-fail if anything is silently skipped. Less ceremony, fewer mistakes, more time on actual design.
