How to Contribute

Workflow for designers proposing new components, tokens, or spec updates.

  1. Propose in Roadmap. Add an entry to the Roadmap with a short rationale. Tag it as proposed.
  2. Get approval. Async review with at least one designer + one developer. Discuss in the Roadmap entry's GitHub issue.
  3. Design in Figma. Build the component variants and states in the UDS Components file. Apply the appropriate stoplight status emoji to the page name (🟠 in progress, 🟡 review, 🟢 production).
  4. Scaffold the per-component folder. Use the new-component Cursor skill (or scaffold by hand) to create uds/components/<component>/ containing spec.json (conforms to spec.schema.json), status.json, changelog.json, impl.json, playground.js, an empty <component>.css, and examples/manifest.json + at least one examples/*.html. Fill in description, whenToUse, whenNotToUse, acceptanceCriteria, props, events, slots, states, accessibility.keyboard, and owner in spec.json at minimum.
  5. Add to the doc site. Add a new <div data-page="..."> in index.html with Examples, Code, Guidelines, Changelog, and Playground tabs. Add a sidebar link. Run bash scripts/aggregate-components.sh so the new id appears in uds/components.json. The Guidelines / Examples / Playground tabs auto-render from the per-component JSON at runtime — no app.js edits needed.
  6. Notify dev. File a Storybook issue requesting the web component implementation. Link to the spec page.
  7. Land in Storybook. Developer implements the web component to match the spec 1:1. Both designer and developer review.
  8. Update status to Production Ready. Once shipped, update current in uds/components/<component>/status.json (and append to history[]) and set the Figma page name emoji to 🟢.
  9. Bump UDS version. If this is a meaningful release, run bash uds-docs/release.sh <new-version> — it snapshots the current uds/ into versions/<old>/uds/, bumps uds/version.json, and refreshes uds/CHANGELOG.json from per-component changelog.json files.

Editing an existing spec

For changes to a component already on the site, edit uds/components/<component>/spec.json directly. The Guidelines tab and spec completeness indicator update automatically. Bump the SITE version, add a SITE_CHANGELOG entry in docs/data/site-changelog.js, append a per-component entry to uds/components/<component>/changelog.json, then run bash scripts/aggregate-changelog.sh if the change is user-facing.

Draft spec workflow

Set "draft": true in a component's spec.json to hide it from the live site. Append ?draft=1 to any URL to view drafts with a watermark.
