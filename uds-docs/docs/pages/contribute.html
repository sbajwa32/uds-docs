How to Contribute

Workflow for designers proposing new components, tokens, or spec updates.

  1. Propose in Roadmap. Add an entry to the Roadmap with a short rationale. Tag it as proposed.
  2. Get approval. Async review with at least one designer + one developer. Discuss in the Roadmap entry's GitHub issue.
  3. Design in Figma. Build the component variants and states in the UDS Components file. Apply the appropriate stoplight status emoji to the page name (🟠 in progress, 🟡 review, 🟢 production).
  4. Create the spec. Add content/<component>.json following the schema. Fill in description, whenToUse, whenNotToUse, acceptanceCriteria, props, events, slots, states, accessibility.keyboard, and owner at minimum.
  5. Add to the doc site. Add a new <div data-page="..."> in index.html with Examples, Code, Guidelines, Changelog, and Playground tabs. Add a sidebar link. The Guidelines tab will auto-render from the JSON.
  6. Notify dev. File a Storybook issue requesting the web component implementation. Link to the spec page.
  7. Land in Storybook. Developer implements the web component to match the spec 1:1. Both designer and developer review.
  8. Update status to Production Ready. Once shipped, update COMPONENT_STATUS in app.js and the Figma page name emoji to 🟢.
  9. Bump UDS version. If this is a meaningful release, bump UDS X.Y in versions.json and add a CHANGELOG entry under the new release.

Editing an existing spec

For changes to a component already on the site, edit content/<component>.json directly. The Guidelines tab and spec completeness indicator update automatically. Bump the SITE version, add a SITE_CHANGELOG entry, and a per-component CHANGELOG entry if the change is user-facing.

Draft spec workflow

Set "draft": true in a component's JSON to hide it from the live site. Append ?draft=1 to any URL to view drafts with a watermark.
