Platform Support

Browser compatibility, performance baseline, and platform notes that apply across the entire system. Components only override these when they have specific exceptions.

Browser compatibility

  • Modern evergreen browsers. Chrome, Edge, Firefox, Safari — last 2 major versions.
  • CSS features required: CSS custom properties, :has(), logical properties, color-mix(), @layer.
  • JS features required: ES2020+, native modules, fetch, Promise, queueMicrotask.
  • No IE / legacy. UDS does not support Internet Explorer or non-evergreen browsers. If a project requires those, it cannot use UDS as-is.

Performance baseline

  • Render budget: a typical UDS page should hit Largest Contentful Paint < 2s on mid-tier hardware.
  • Bundle size: uds.css is the only required CSS. Component CSS is loaded via @import.
  • JS overhead: uds.js is the orchestrator that lazy-imports per-component JS as needed. Vanilla HTML pages need it; framework projects (React/Vue web component consumers) do not.
  • Components with size considerations: Data Table should virtualize at 200+ rows; Dialog should lazy-mount. See per-component Performance notes for any other exceptions.

Web component lifecycle (Storybook)

  • Each Storybook web component is framework-agnostic — register the custom element once, then use the tag in any project (React, Vue, vanilla, Lit, etc.).
  • Components must implement connectedCallback for setup and disconnectedCallback for teardown (event listeners, observers).
  • Reflected attributes: variant, size, disabled, open, selected. Property changes update the attribute and vice versa.

Server-side rendering

  • UDS CSS works in any SSR setup (Next.js, Nuxt, Astro, Rails, etc.) — it's plain CSS with no runtime.
  • Storybook web components hydrate client-side. For SSR-critical content, output the HTML structure server-side using UDS classes; the web component will progressively enhance.
