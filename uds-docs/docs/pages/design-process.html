Design Process

How a UDS component moves from idea to production. First artifact in a longer design-process documentation effort. The bottom half of this page weighs the proposal against industry best practices.

forum
This is a discussion document, not a live process. The current process inside the repo (status enum, sync skills, banner copy, etc.) has not been changed. This page captures a proposed design process and a side-by-side review against industry best practices. Nothing on this page is in effect until adopted as a deliberate change.

The stoplight system on this site marks how far each component has moved through the design process. The proposal below tightens what each color means, who owns it, and what is true at each stage. The recommendations section then weighs the proposal against how mature design systems (Polaris, Carbon, Lightning, Spectrum, Material) handle the same problem.

Part 1 — The proposed process

The model

The stoplight is the designer's progress system. It tracks how a component moves from idea to a final, dev-handoff-ready design contract. Storybook implementation is a separate dev workstream, gated by the Green status but otherwise outside the stoplight. The process splits into two universes — Figma for design in motion, the doc site for the shipped contract — with a handoff in the middle.

Figma — design in motion

Component is being designed: shape ideas, anatomy, variants, states. No uds-docs/uds/components/<id>/ folder, no sidebar entry, no status.json value. Figma page-name prefix is the system-of-record.

Red — Exploration Orange — In Progress

Doc site — shipped contract

Component has crossed into a written contract: spec.json, examples, sidebar entry, status.json value. Green = the contract is final, locked, and signals to devs that implementation can begin.

Yellow — In Review Green — Production Ready Black — Deprecated

Lifecycle at a glance

Red → Orange → Yellow → Green ⤳ Black codeStorybook (dev) — gated by Green Forward path is solid; sunset to Black is dotted. Backward moves (Yellow → Orange, Green → Yellow) are covered under "Iteration & change" below.

Stage checklists

Each stage is a card with three things: what it means, what is true while at this stage, and what must be checked off to advance.

Red — Exploration

Universe: Figma Key: exploration Owner: Designer

First stab on the canvas — shape ideas, anatomy candidates, rough variants. Not yet a coherent component direction.

While at this stage
  • Figma page exists with the red prefix
  • At least one frame on the canvas captures a first design attempt
  • Design questions / constraints being explored are noted on the page
  • No doc-site folder exists for this component yet
To advance to Orange
  • One design direction is chosen from the exploration alternatives
  • Exploration alternatives are archived or annotated (history preserved)
  • Figma page is renamed with the orange prefix

Orange — In Progress

Universe: Figma Key: in-progress Owner: Designer

Designer has committed to a direction and is building it out: real variants, real states, token-bound.

While at this stage
  • Figma page has the orange prefix
  • Component set is drafted with variant properties (e.g. size, variant, state)
  • Key interactive states are defined (default, hover, focus, active, disabled — plus any component-specific states)
  • All colors / spacing / fonts are token-bound (no hardcoded values)
  • Component anatomy is labeled in Figma (annotations for each named part)
  • Still no doc-site folder for this component
To advance to Yellow
  • Design feels settled enough to write a final spec against
  • Figma page is renamed with the yellow prefix
  • new-component skill is triggered: doc-site folder scaffolded under uds/components/<id>/
  • Initial status.json created with current: "review"

Yellow — In Review

Universe: Doc site Key: review Owner: Designer

First doc-site stage. Designer is authoring the spec contract — the written description that devs will eventually implement against. No dev involvement at this stage.

While at this stage
  • spec.json exists with at least: description, whenToUse, whenNotToUse, anatomy named in slots[], accessibility.keyboard[], acceptanceCriteria[], owner
  • At least one example HTML in examples/
  • Component page renders on the doc site with Examples / Code / Guidelines / Changelog / Playground tabs
  • "Not production-ready" banner is visible at the top of the component page
  • Sidebar status badge shows yellow
  • Spec X/22 completeness pill is climbing toward 80%
  • Peer-designer review happens here (optional but recommended)
To advance to Green
  • Spec X/22 completeness is ≥ 80%
  • Spec is locked from the designer's POV — no further changes pending
  • Figma page renamed with the green prefix
  • status.json current set to production; since set to the current UDS version
  • history[] updated with the transition
  • Per-component changelog.json gets a { type: "changed", text: "Status: review → production" } entry

Green — Production Ready

Universe: Doc site Key: production Owner: Designer

Spec is locked. The act of flipping to Green is the dev-handoff signal — devs can now implement against a stable contract.

While at this stage
  • status.json current: "production"
  • "Not production-ready" banner is removed from the component page
  • Sidebar status badge shows green
  • Figma page has the green prefix
  • Component is consumable as a stable contract
What happens off-stoplight
  • Devs see the Green status and begin Storybook implementation (parallel workstream)
  • Spec refinements from dev feedback land via the per-component changelog as normal evolution
  • New product work can build against the spec
To deprecate (advance to Black)
  • Replacement component identified (and ideally at Production Ready itself)
  • whenNotToUse is updated to point at the replacement
  • status.json current set to deprecated
  • Figma page renamed with the black prefix
  • Per-component changelog.json gets a { type: "deprecated" } entry
To regress to Yellow (rare — major redesign only)
  • Explicit user direction (requires confirmation per the sync-figma-component-status skill)
  • Reason documented in the changelog

Black — Deprecated

Universe: Doc site Key: deprecated Owner: Designer

Sunsetted. Preserved for back-compat consumers; new use is discouraged.

While at this stage
  • status.json current: "deprecated"
  • whenNotToUse documents the replacement
  • Figma page has the black prefix
  • Component remains visible on the doc site for back-compat consumers
  • Terminal state — no forward progression
Maintenance
  • Existing consumers can keep using
  • New consumers are directed to the replacement
  • Component eventually gets removed in a major UDS release (separate destructive operation, requires user direction)

Iteration & change

Components don't move monotonically forward. Three patterns handle the real-world cases.

1. In-stage edits (most common)

Routine changes — spec wording tightens, an example refines, a Figma node gets cleaner, a token reference swaps for a better one. Stay at the current status.

Steps: edit the relevant file → add a per-component changelog.json entry naming what changed and why → if Figma was the source, run the figma-component-inspector subagent and the sync-figma-component-spec skill → bash scripts/aggregate-changelog.sh → standard SITE bump + commit + push.

No status flip. The "Not production-ready" banner stays up at Yellow; the Green badge stays at Green.

2. Backward moves (intentional)

Sometimes design surfaces a question the current stage can't answer.

Yellow → Orange: spec authoring revealed the Figma design isn't actually settled. Back to Figma for more work. Flip the Figma page prefix to orange; status.json current to in-progress; changelog entry naming the reason. Doc-site folder stays (banner stays up).

Yellow → Red: rare. Want to explore alternatives. Same shape, status to exploration. Doc-site folder stays.

Green → Yellow: major redesign. The only backward move that requires explicit confirmation per the sync-figma-component-status skill (the others are designer-call). Banner reappears.

history[] in status.json is the audit trail — every status flip appends, so back-and-forth is fully preserved.

3. Drift discovered later

Figma moved but the doc site didn't (or vice versa).

Run the figma-component-inspector subagent → inspector reports doc-site / Figma mismatches with confidence + risk classifications → sync-figma-component-spec applies high-confidence non-breaking changes automatically; potentially-breaking changes stop for user direction → per-component changelog gets an entry per logical change. Status only flips if the designer explicitly decides it needs to.

Default when in doubt

Stay in-stage and add a per-component changelog entry. Backward status moves are real but should be intentional, not reflexive. The audit-changelog-currency.sh audit enforces that meaningful edits get a changelog entry — so the system catches you if you forget.

Where the status lives

  • Figma page-name prefix — all 5 stages (red / orange / yellow / green / black). The canvas-side signal.
  • uds-docs/uds/components/<id>/status.json — only exists from Yellow onward. Holds current (the status key) + since (the UDS version it entered) + history[] (audit trail).
  • Sidebar status badge — only renders for components that have a doc-site folder (Yellow onward). Color follows the status key.
  • "Not production-ready" banner — visible on Yellow component pages; removed at Green; not applicable to Black (since Black components are intentionally still consumable for back-compat).
Part 2 — Best-practices review

How mature design systems handle this

Looking at Polaris (Shopify), Carbon (IBM), Lightning (Salesforce), Spectrum (Adobe), and Material (Google): all of them use 3–4 lifecycle stages, most have an explicit "this might still change" stage between Review and Locked, and none publicly track pre-design exploration as a system state. Five recommendations follow.

Recommendation 1 · Highest impact

Add a Beta / Experimental stage between Green and "forever"

Almost every mature design system has a stage that says to devs and consumers: "you can use this, but the spec may still change." Examples:

  • Polaris: Stable / Experimental / Legacy / Deprecated
  • Carbon: Experimental / Draft / Stable / Deprecated
  • Lightning: Drafted / Beta / Released / Deprecated
  • Spectrum: Working draft / Beta / Released

The current proposal jumps from Yellow (private review) to Green (locked contract), then quietly relies on the per-component changelog to absorb spec evolution after Green. Internally coherent, but it doesn't match what devs from any other design system would expect. They'll see Green and assume "stable forever" — then be surprised when the changelog adds breaking changes.

What this would look like

Option A — a sixth status:

exploration → in-progress → review → experimental → production → deprecated

Option B — a flag on Green:

status.json gets a stability field: "beta" or "stable". Green badge shows a small "Beta" pill until stability is upgraded.

Either way: devs implement against Beta knowing the spec may shift. Green (or "Stable") stays for the components that are genuinely locked.

Recommendation 2

Drop Red (Exploration)

No major design system publicly tracks exploration as a system stage. It happens in a designer's working file and stays invisible until it's coherent enough to be Orange (or whatever the equivalent is). Tracking exploration in the public system adds noise: visitors see "29 components" but most are exploratory, which makes the system look bigger and less mature than it is.

What this would look like

The Figma file would still have pages, but Red prefixes (and Exploration as a tracked status) would go away. Pre-Orange work lives in private Figma frames or working files. The system only starts tracking a component when it reaches Orange (a real direction with variants).

Stages collapse to in-progress → review → production → deprecated (4 stages), which lines up well with Polaris and Carbon.

Recommendation 3

Add a Storybook checklist item to Green

Most successful systems treat "shipped to consumers" as the union of spec + implementation. Carbon, Polaris, and Lightning all require the implementation to exist before they call something Stable. The proposal says Storybook is owned by devs, gated by Green but otherwise off-stoplight. That could create the same Figma / code drift problem we already see, just relocated.

What this would look like

The Green stage card would add one checklist item to "While at this stage":

  • Storybook implementation exists and matches the spec

It doesn't have to gate the stage flip (designer can flip to Green to signal "ready for dev"), but it must be tracked. A small audit or a "Storybook: linked / unlinked" indicator on the component page would make the relationship visible.

Recommendation 4

Require a reason field on backward moves

Successful systems make backward moves intentional and slightly painful (a reason field, a peer review, a publicly-visible regression note). The current proposal is designer-call with no friction beyond a changelog entry. That risks ping-ponging — Yellow → Orange → Yellow → Orange — which is rarely productive.

What this would look like

Every backward move appends a history[] entry that includes a reason field, surfaced on the component page as a visible "Why this regressed" note.

history[] entry shape becomes:

{ "version": "0.3", "status": "in-progress", "reason": "Variants for icon-only need rework" }

Visibility creates the discipline. Designers who know the reason will be public are more likely to think carefully before flipping back.

Recommendation 5

Add a "How this differs from typical design systems" note

The two-universe model (Figma owns half the lifecycle; doc site owns the other half) is novel. Most successful systems have one universe — the doc site is the spec, code is the implementation, the Figma file is a downstream artifact (or even auto-generated).

What this would look like

A short callout on this page (and on the Getting Started page) acknowledging the divergence:

"Most design systems treat the documentation site as the single source of truth. UDS splits responsibility: Figma owns design-in-motion (Red / Orange), the doc site owns the shipped contract (Yellow / Green / Black). This is a deliberate choice for a small team where Figma is where design actually happens — but if you're coming from Material, Polaris, or Carbon, expect to look in two places during the early stages of a component."

If all five recommendations were adopted

The lifecycle would look like this — 4 stoplight stages with a Beta sub-flag on Green, dropped Red, Storybook checklist on Green, reason-tracked backward moves:

Orange — In Progress → Yellow — In Review → Green — Production Beta ⤳ Black — Deprecated Pre-Orange work happens in private Figma files (off-system). Storybook implementation is tracked on the Green stage's checklist. Backward moves require a reason field.

Concrete changes from the Part 1 proposal:

  • Status enum: in-progress | review | experimental | production | deprecated — five values, two of which co-exist on Green (production with stability: "beta" or just experimental as a separate status).
  • Stoplight badges: 4 forward colors instead of 5. Beta surfaces as a small overlay pill on the Green badge, not its own color.
  • Figma prefixes: drop the red prefix. Component pages enter the system at orange.
  • Sidebar: a Beta indicator next to components that are Green but not yet Stable, so consumers can tell at a glance.
  • Green stage checklist: adds "Storybook implementation exists and matches the spec."
  • history[] entries: backward moves require reason. Surfaced on the component page as "Why this regressed."
  • This page (and Getting Started): adds the "How this differs from typical design systems" callout.
